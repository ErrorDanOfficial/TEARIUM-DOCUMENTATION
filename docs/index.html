1. Design Goals

1.1 Core Objective

TEARIUM OS is built to provide an independent terminal-based virtual platform inside Teardown. It supports singleplayer and multiplayer execution with a persistent software layer that doesn't rely on Workshop updates.

1.2 Decoupled Content Model

The mod provides the platform, not the content. The Workshop version remains static, while all software (games, runtimes) is user-created, locally installed, or distributed via multiplayer.

1.3 Distribution Model

A decentralized ecosystem where any player can develop, install, and share software without a central repository or mod updates.

2. System Architecture Overview

Modular Kernel Architecture components:

  • Kernel (main.lua): Interpreter, scheduler, loader, networking.
  • CPU Emulator (script/cpu.lua): 16-bit virtual processor emulator.
  • Display Driver (script/screen.lua): Character renderer (40x16 grid).
  • Storage Driver (persistence.lua): Registry VFS system.
  • Runtime System (runtimes/): Dynamically loaded libraries ($G.require).

3. Execution Model

Single-threaded, event-driven VM using cooperative multitasking.

3.1 Execution Flow

  1. Teardown update loop calls Kernel.
  2. Kernel: processes input, timers, networking.
  3. Active coroutine is resumed until it yields.
  4. Control returns back to Kernel.

3.3 Event Queue

Maintains a FIFO queue for: key, key_up, char, timer, CC_TICK, and custom events.

4. Hardware Emulation - TEARIUM CPU

16-bit address space (64 KB). Registers: regA, regB (8-bit), pc, sp (16-bit). Flag: zeroFlag.

0x0100-0x01FF : Buffer
0x1000-0x1FFF : GFX VRAM
0x2000-0x23FF : Text VRAM
0x3000-0x33FF : FG Colors
0x4000-0x43FF : BG Colors
0x8000-0xFDFF : User Code
0xFE00 : Keyboard Input

Tools & Automation (Powershell)

sync_saves.bat

Synchronizes VFS data between Singleplayer and Multiplayer sandboxes. Critical for cross-version play.

build_vfs.ps1

Encodes files to vfs_assets.lua format. Scans project folders and generates master RAM tables.

restore_vfs.ps1

Extracts registry data back to real files on disk.

5. Display System

Memory-mapped character rendering (40x16 grid) using a 5x7 bitmap font. Optimized strategy: 640 quads per frame maximum.

6. ComputerCraft Compatibility Layer (cc_core)

Injects standard APIs into _G: term, fs, os, parallel, bit, paintutils.

  • term: Direct VRAM mapping, scrolling support, cursor blink.
  • window: Fully implemented windowing system for Pine3D engines.
  • os: pullEvent() yields coroutine; clock() uses Teardown time.
  • fs: RAM -> Registry -> MOD/Disk resolution pipeline.

7. Virtual File System (VFS)

Stored in AppData/Local/Teardown/savegame.xml. Uses chunked storage (4KB-8KB) to bypass XML limitations.

7.3 mem_vfs Cache

In-memory representation for fast access and avoids repeated Base64 decoding.

8. VFS Bundler

Bypasses Teardown's "no-io" sandbox. Encodes CCDoom-master and runtimes into Base64 strings. Automated via build_vfs.ps1.

9. Program Lifecycle

BOOT Command Flow: Locate binary -> Decode -> Parse @runtime headers -> Load dependencies -> Inject APIs -> loadstring() -> Create Coroutine -> Start Execution.

10. Runtime System

10.1 Metadata Headers

Uses script comments: @title (Name), @runtime (Modules), @include (Files), @bundle (Assets).

10.2 Dual-Layer Architecture

  • Native Lua Hub: Runs at full PC speed. Use for Doom and 3D.
  • Virtual CPU: 16-bit simulation for retro experiments.

11. Networking and Distribution

Client-as-source P2P system. Receiver confirms via [Y/N] prompt.

SHARE: Recursively bundles main game, all @includes, @bundles, and required @runtimes before transfer.

12. CLI Package Management

  • LS: Smart list with dependency counts and filters (-A for admin view).
  • INSTALL: Atomic registry write into VFS.
  • RM: Recursive cleanup (use -rt to keep runtimes).
  • History: Up/Down arrows to cycle through previous commands.

13. Keyboard Shortcut Reference

F11 : Toggle Interaction Mode
ESC : Exit Interaction Mode
F9 : Toggle Terminal Overlay Visibility
F8 : Force Process Termination (Kill)

14. Developer Workflow

  1. Write Lua -> 2. Add headers -> 3. Bundle (PS1) -> 4. INSTALL in SP -> 5. SHARE in MP.

15. Practical Programming Model

Standard event-loop using os.pullEvent(). Supports parallel.waitForAny() for multitasking.

16. System Constraints

Display 40x16 (VRAM limit). Cooperative multitasking (Must yield). XML-based storage.

17. Final System Definition

Persistent, event-driven OS embedded in Teardown, combining Lua execution and P2P distribution.

18. Key Outcome

Independent software layer. No dependecy on mod updates. Extensible architecture.

19. Recent Updates (Stabilization)

20. Developer Quick Start

How to create games: write standard Lua code with a header.

-- @title My Game
-- @runtime cc_core

term.clear()
while true do
  local ev = os.pullEvent()
  term.write("FPS: " .. os.clock())
end
